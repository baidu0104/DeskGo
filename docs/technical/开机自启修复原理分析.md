# DeskGo 开机自启失效问题（Win11/MSIX环境）分析与修复报告

## 1. 问题背景
在 Windows 11 环境下，通过 MSIX 包安装应用后，用户在软件的托盘菜单中勾选“开机自启”时，并未达到预期的系统启动效果。而当软件以纯绿色免安装方式运行在某些环境下时，行为可能有所差异。

## 2. 第一性原理剖析（Root Cause Analysis）
开机自启的本质：操作系统在启动（或用户登录）时，通过读取特定的系统清单（如注册表中的 `Run` 键或开机启动文件夹），寻找需要自动执行的程序路径并唤起。

### 2.1 失效的代码逻辑
在修复前的 `src/core/configmanager.cpp` 源文件中，管理着开机自启的具体落实操作：
```cpp
void ConfigManager::updateAutoStartRegistry(bool enabled)
{
#ifdef Q_OS_WIN
    bool isPackaged = false;
    // 动态检查是否运行在 MSIX 包容容器环境 (基于 GetPackageFamilyName)
    // ...
    if (isPackaged) {
        // [原意]：MSIX 应用无法直接走传统注册表方法自启，需要通过 WinRT StartupTask API。
        // [现实]：因为当前项目属于 C++ Qt (MinGW编译) 体系，未接入 WinRT 投影库，
        // 故此处仅留下了一段注释，随后直接执行了 return; （跳过后续的写入逻辑）。
        return; 
    }
    // ...（后续的注册表传统读写被跳过）
#endif
}
```
**致命点**：在 Windows 11 (MSIX) 中运行，`isPackaged` 被确认为 `true`，触发了 `return`。代码没有执行任何操作系统级别的通知，仅自嗨式地修改了软件内存里的布尔值并保存到了 `fencing_config.json` 里。系统自然不会允许其开机启动。

### 2.2 UWP / MSIX 扩展清单的冗余冲突
在项目的原始设计中，伴随着一段“半透明”的想法，`AppxManifest.xml` 中加入了下面这段启动项拓展：
```xml
<desktop:Extension Category="windows.startupTask" Executable="DeskGo.exe" EntryPoint="Windows.FullTrustApplication">
    <desktop:StartupTask TaskId="DeskGoStartupTask" Enabled="false" DisplayName="DeskGo" />
</desktop:Extension>
```
按照微软的规范，这段声明确实是让 MSIX 具备加入任务管理器“启动”选项卡的资质，但它有一个硬性前置条件：**需要 C++ 或 C# 代码主动调用 Windows.ApplicationModel.StartupTask 相关接口动态申请用户的权限。**
既然我们的 C++ 代码目前无法简易地调用该接口，该 XML 节点的存在不仅是死节点，对于系统虚拟化注入注册表时还会带来优先级或状态重置的潜在干扰（因为声明中写了 `Enabled="false"`，也就是预设就是禁止，且由于没有代码去修改它，它将永远处于死锁禁用状态）。

## 3. 极简修复方案与兼容性原理（KISS 原则）
抛弃过度封装与错综复杂的 WinRT 交叉调用。

**事实依据**：
微软引入 Desktop Bridge (桌面桥接) 的主要目的就是向下兼容海量的经典 Win32 应用程序。在现代 Windows 10/11 的 MSIX 容器环境中，**底层的注册表（Registry）对打包应用提供了全方位的虚拟化（Virtualization）支持。**

当一个被 MSIX 封装的经典 Win32 应用程序（如我们的 Qt DeskGo）直接去注册表中写入：
`HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\CurrentVersion\Run` 时：
1. **沙盒层拦截**：系统的 MSIX 沙盒驱动会捕获该写入动作。
2. **正确映射**：沙盒并非阻止它，而是将这笔写入重定向到了宿主操作系统对应的现代应用启动项配置中。
3. **静默生效**：应用程序会被正常添加到 Windows “任务管理器” 的 “启动” 页面中，受用户的统一管控，且能够如期跟随系统自启。

### 3.1 实施步骤
1. **拆除包容检测墙：**
   全面删除 `configmanager.cpp` 中的 `isPackaged` 检测逻辑与导致直接退出的 `return;` 分支。

2. **恢复纯粹的注册表写盘：**
   所有环境下，统一通过 `RegSetValueExW` 写入 `Run` 键。无论是未来的纯净绿色版，还是应用商店 MSIX 版，此套逻辑能借助操作系统自带的虚拟化底座达到完美的同构兼容，消除了代码中的“环境歧视”。

3. **去除清单死节点：**
   删除 `AppxManifest.xml` 中的 `<desktop:Extension Category="windows.startupTask" />` 声明块，解除打包平台对自启功能的冗余接管权卡扣。

## 4. 结论
面对所谓的新型运行环境（如 MSIX/Appx），避免过度防御性编程（如因为觉得没权限所以干脆不去执行）。从操作系统的系统级封装原理出发，直接复用成熟的 Win32 API 绝大部分情况下都能获得原生级的映射支持，不仅代码精简且免除了不可预知的异步权限问题。
